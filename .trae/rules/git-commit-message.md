# 整体要求
git 的 message 信息需要是总结性的表述，不宜过长

且你需要满足如下的详细规则要求

## 内容要求
因为项目 git 提交无非是：
- 新增某种功能
- 改变了某种功能
- 删除了某种功能
- fix 或者优化了某种功能

因此如果此次 git 有新增功能，或改动功能，或删除功能，或则 fix/优化功能，你需要写上对应的总结

内容都需要是总结性的，不宜过长！

## 格式要求
meesage 格式如下，但是不一定 `1,2,3,4` 4 项都存在，得看代码实际的改动内容
```md
整体总结性的描述
1. 新增功能总结（如果存在的话）
2. 改动功能总结（如果存在的话）
3. 删除功能总结（如果存在的话）
4. fix 或优化功能总结（如果存在的话）
```
并且 message 内容展示的优先级上不一定按照上方的 `1,2,3,4` 的顺序来，你需要自己判断代码主要的变动内容，识别最重要最主要的代码变动，按照重要性来 `1,2,3,4` 的顺序来展示
